Getting Started

Installation

OpenSynapse can be installed three ways depending on your environment.

Desktop App (Windows, macOS, Linux)

  1. Download the installer for your platform from the releases page.
  2. Run the installer. On macOS, drag the app to /Applications. On Windows, run the .msi installer. On Linux, use the .deb or .AppImage.
  3. Launch OpenSynapse. The desktop app bundles the control plane, web UI, and a local SQLite database — everything runs on your machine with no external dependencies.

Docker (Self-hosted Server)

  1. Ensure Docker and Docker Compose are installed on your server.
  2. Clone the repository or download the deploy/docker/ directory.
  3. Run the stack: 
    cd deploy/docker
docker compose up -d
  4. Open http://localhost:8080 in your browser. The Docker setup includes the Go control plane (port 8090) and the React web app served via nginx (port 8080), which also proxies API and WebSocket requests.

Kubernetes (Cluster Mode)

  1. Install the Helm chart from deploy/helm/: 
    helm install opensynapse ./deploy/helm \
  --set postgres.enabled=true \
  --set s3.endpoint=your-minio-endpoint
  2. Configure your ingress to expose the web UI.
  3. For distributed test execution, deploy the k6-operator in your cluster. OpenSynapse will automatically detect it and enable cluster mode.

First Launch

When you open OpenSynapse for the first time, you'll see the Home view. This is your starting point for everything.

UI Overview

The interface is organized around these primary views:

  • Home — Quick actions: New Test, Crawl Application, Playground. Recent plans and recent runs.
  • Plans — All your test plans. Click one to open the Plan Builder.
  • Runs — History of all test executions with status, duration, and key metrics.
  • Playground — Postman-style endpoint explorer for ad-hoc HTTP requests.
  • Settings — AI provider configuration, appearance, integrations, and storage.

Navigation

Use the top navigation bar to switch between views. Each view remembers your scroll position and filter state. The browser's back and forward buttons work naturally since all views are URL-routed.

Tip: OpenSynapse is designed so you can go from installation to running your first load test in under 10 minutes. Follow the next section to create your first test.

Creating Your First Test

  1. From the Home view or the Plans list, click the "New Test" button.
  2. The Template Gallery opens as a modal. You'll see 9 templates, each with an animated load curve preview showing the traffic pattern.
  3. For your first test, choose "Load" — it ramps to 50 virtual users over 2 minutes, holds for 10 minutes, then ramps down. This is a safe, typical test pattern.
  4. Click "Use template". A new plan is created and the Plan Builder opens automatically.
  5. The plan already has a root Scenario node configured with the Load template settings. Click the Scenario node to edit its properties in the right panel.
  6. Add an HTTP Request node as a child of the Scenario. Set the method to GET and enter your target URL.
  7. Click the "Run" button in the toolbar to start the test. You'll be taken to the live Run View.

Note: You can also start with a Blank Plan if you prefer to build everything from scratch. The blank option appears at the top of the Template Gallery.

Alternatively, if you just want to test a single endpoint quickly, use the Playground instead — no plan needed.

Building Test Plans

The Three-Pane Layout

The Plan Builder uses a three-pane layout:

  • Left pane (Node Tree, 280px) — A hierarchical, collapsible tree of all nodes in your plan. Supports drag-and-drop reordering and search filtering.
  • Centre pane (Flow Canvas) — A visual flow diagram showing the selected branch as connected nodes. Powered by React Flow.
  • Right pane (Property Panel, 280px) — The form editor for the currently selected node's properties. Switches to the Fragment Library when the Fragments button is active.

Node Types

OpenSynapse has 14 node types organized into three categories:

Samplers

Nodes that generate requests or execute code during the test.

H HTTP Request

Send an HTTP request with configurable method, URL, headers, body, and redirect behavior. Supports variable interpolation with ${varName}.

W WebSocket

Open a WebSocket connection with configurable messages to send, expected responses, connect timeout, and disconnect behavior.

{ Code Block

Write custom JavaScript that runs in the k6 VU context. Use for anything the visual builder can't express.

Controllers

Nodes that control the flow of execution. All controllers can have children.

S Scenario

Defines an executor (constant-vus, ramping-vus, constant-arrival-rate, etc.) and its parameters. Children run under this executor.

? If

Children run only when the JavaScript condition evaluates to true.

: Else

Must follow an If node. Children run when the If condition was false.

L Loop

Repeat children a fixed number of times. Default: 5 iterations.

T Transaction

Group children and report aggregated metrics. Translates to k6 group().

1 Once Only

Children run once per VU regardless of iteration count. Ideal for login and setup steps.

R Random

Picks one child at random per iteration. Optional weights per child.

Auxiliary

Support nodes that configure data, timing, and validation.

A Assertion

Validate responses: status code, body content (substring, regex, JSONPath), header values, or response time thresholds.

Z Timer

Insert think time after a request. Types: constant (default 1000ms), uniform random, or Gaussian.

D Data Source

Feed test data from CSV files, JSON files, or inline arrays. Rows distribute across VUs via k6 SharedArray.

E Environment

Bind an Environment entity to scope variables to the plan.

Adding Nodes

There are three ways to add a node to your plan:

  • Context menu: Right-click on a parent node in the tree and select the node type from the menu.
  • Add button: Click the "+" button that appears in the tree when hovering over a node.
  • Drag from palette: Drag a node type from the palette into the tree at the desired position.

You can also cut, copy, paste, and duplicate nodes — even between different plans.

Code View

Click "Show Script" in the toolbar to see the generated k6 JavaScript. This is a read-only preview of the exact script that will run when you execute the test. Click "Close" to return to the flow canvas.

Search

Type in the search box at the top of the node tree to filter nodes. Search matches node names, types, and property values. Matching nodes and their ancestors remain visible; everything else is hidden.

Undo / Redo

All editing operations support undo and redo. Use the buttons in the toolbar or the keyboard shortcuts Ctrl+Z and Ctrl+Y.

Auto-save

Every property change is saved automatically (debounced 300ms). The toolbar shows the save status: Saving..., Unsaved, or Saved. Plans are versioned so you can browse history and revert.

Running Tests

Starting a Run

From the Plan Builder, click the "Run" button in the top-right of the toolbar. The plan is compiled to a k6 script and execution begins. You're automatically navigated to the Run View.

Warning: Make sure your plan has at least one Scenario node with an HTTP Request child before running. A plan with no requests will complete instantly with no data.

Live Control

While a test is running, the Live Control Panel appears as a sidebar on the right side of the Run View. This is OpenSynapse's signature feature: the ability to adjust test parameters in real time without restarting.

How it works: Live control uses k6's externally-controlled executor combined with the xk6-kv extension for a token bucket. Changes are applied within seconds and logged as events on the run timeline.

VU Slider

Adjust the number of virtual users from 0 to 500. The current active VU count is displayed at the top. Drag the slider or type an exact number, then click "Apply" to send the change.

RPS Slider

Control the requests-per-second rate from 0 to 10,000 (step: 10). The current measured RPS is shown. The token bucket in xk6-kv is updated when you apply.

Duration Control

See the elapsed time and adjust the total test duration. Use the "+5 min" and "-5 min" quick buttons, or type an exact duration in seconds and click "Apply".

Viewing Results

Post-Run Report

After a test completes, the Run View remains showing the final state of all metrics and charts. The charts render from the persisted dataset at full fidelity (not just the live samples).

Summary Statistics

The six metric cards show the final summary values:

  • Total Requests — The total number of HTTP requests made during the run.
  • Failed — Count of requests that returned non-2xx status codes or timed out.
  • Error Rate — Percentage of failed requests.
  • Throughput RPS — Average requests per second over the entire run.
  • p95 Response Time — 95th percentile response time in milliseconds.
  • Peak VUs — Maximum number of active virtual users during the run.

Export Options

Click the "Export" dropdown in the toolbar to download the results in one of three formats:

Comparing Runs

Selecting Runs to Compare

  1. Navigate to the Runs list view.
  2. Click the checkbox next to each run you want to compare (up to 5 runs).
  3. Click the "Compare" button that appears in the toolbar.

Comparison View

The comparison view overlays metrics from all selected runs on shared charts. Each run gets a distinct colour and appears in the chart legend.

Summary Block

At the top, a summary shows:

  • Which runs are being compared (IDs and dates)
  • Which metrics improved (highlighted in green)
  • Which metrics degraded (highlighted in red)
  • Absolute and percentage differences

Detection Threshold

Changes smaller than 5% are not flagged as improvements or degradations to avoid noise from natural variance. Only statistically meaningful differences are reported.

Time Normalization

Differences are computed per second of elapsed time within the shorter run. You can switch to normalized mode (0-100% progress) if the runs have different durations and you want to compare equivalent phases.

Tip: Compare a Smoke test run against a Load test run to understand how your application behaves under minimal vs. typical traffic.

Endpoint Playground

The Playground is a Postman-style HTTP client built into OpenSynapse. Use it to explore APIs, test individual endpoints, and save requests for later use in test plans.

Building a Request

  1. Select the HTTP method from the dropdown: GET, POST, PUT, PATCH, DELETE, HEAD, or OPTIONS. Each method is colour-coded (GET = teal, POST = violet, PUT/PATCH = amber, DELETE = red).
  2. Enter the URL in the address bar.
  3. Configure the request using the four tabs below:
    • Params — Query parameters as key-value pairs.
    • Headers — Request headers with enable/disable toggles.
    • Body — Choose None, Raw, JSON, or Form. A text area appears for the body content.
    • Auth — Authentication configuration (see below).
  4. Click "Send" (or press Ctrl+Enter) to execute the request.

Crawler & OpenAPI Import

The crawler automatically discovers and maps your application's endpoints, then generates a draft test plan from the results.

Starting a Crawl

  1. Click "Crawl Application" from the Home view.
  2. Configure the crawl settings:
    • Entry URL — The starting page (e.g., https://myapp.example.com).
    • Crawl Engine — Choose the engine that best fits your target:
      • Rod (Browser) — Headless Chromium via DevTools Protocol. Renders JavaScript, captures all network requests (XHR, fetch, document loads), and handles form login. Best for SPAs and JavaScript-heavy applications.
      • Colly (HTTP) — Pure Go HTTP crawler. Fast and lightweight with zero browser dependencies. Discovers links via HTML parsing. Best for static HTML sites and API documentation. Cannot render JavaScript.
      • ZAP (Security) — Integrates with an OWASP ZAP sidecar via its REST API. Security-focused crawling with built-in vulnerability detection. Requires the ZAP container: docker compose --profile security up.
    • Auth — Optional: form login (username/password), bearer token, or basic auth.
    • Depth — How many link levels to follow (default: 3).
    • Same-origin only — Stay on the same domain (default: on).
    • Path blocklist — URLs to skip (default includes /logout, /delete, and DELETE methods).
    • Request limit — Safety cap on total requests (default: 500).
  3. Click "Start Crawl". The selected engine navigates your site and captures traffic.

What the Crawler Captures

For every page visited, the crawler records:

  • All network requests (XHR, fetch, document loads) with method, URL, headers, body, and response.
  • Links discovered from <a href> tags, navigation buttons, and form submissions.
  • Correlation hints: values that appear in one response and are used in a subsequent request (CSRF tokens, session IDs, JWTs, numeric IDs).

Plan Generation

After the crawl completes, OpenSynapse generates a draft test plan:

  • Requests are grouped by endpoint prefix into Transaction controllers representing logical flows.
  • Correlated values become variable extractors and parameters automatically.
  • Login forms become Once Only controllers with HTTP samplers.
  • The generated plan opens in the Builder for your review and editing.

OpenAPI / Swagger Import

If the target exposes an OpenAPI or Swagger document, the crawler detects it automatically by probing common paths:

  • /swagger.json
  • /openapi.json
  • /v3/api-docs
  • /api-docs

When found, each API operation becomes an HTTP sampler node. Example values from the schema populate request parameters, and security schemes map to auth helpers.

Re-crawl and Diff

You can re-run the crawler on the same target and compare results. The diff view highlights:

  • Added pages and endpoints
  • Removed pages and endpoints
  • Changed endpoints (URL or method changes)
  • Changed response shapes

Choose which changes to merge into your existing plan.

AI Analysis

AI analysis is opt-in and uses your own API keys. No data is ever sent to an AI provider without your explicit click.

Setting Up

  1. Open Settings and go to the AI tab.
  2. Select your provider: OpenAI, Anthropic, or Azure OpenAI.
  3. Paste your API key. It's validated with a minimal test call and stored securely in the OS keychain.
  4. Choose a model from the dropdown (sensible defaults are pre-selected per provider).
  5. Optionally set a monthly spend cap to limit costs.

Analysing a Run

  1. Open a completed run in the Run View.
  2. Click "Analyse with AI" in the toolbar.
  3. A prompt preview shows exactly what data will be sent: the run summary and extracted metrics. No raw request/response data is included — only aggregate metrics.
  4. Confirm to send the request. The AI response appears as markdown in the Insights Panel below the charts.

Built-in Questions

Three standard analysis prompts are available:

  • "What does this run tell me?" — Summary analysis of a single run.
  • "What changed between these runs?" — Comparative analysis in the comparison view.
  • "What test should I run next?" — Follow-up suggestions based on results.

You can also type free-form questions that are sent with the run context.

Cost Tracking

Each analysis shows the token count used and an estimated cost. This helps you stay within your budget and understand how much each query costs.

Tip: AI analysis is most valuable after Stress or Breakpoint tests where there's a clear inflection point. The AI can often identify the VU count or RPS level where performance degraded and suggest possible causes.

Import & Export

JMeter .jmx Import

Migrating from JMeter? OpenSynapse can import .jmx files and convert them to native test plans.

  1. Go to Plans and click "Import".
  2. Select your .jmx file.
  3. The importer parses the XML, maps JMeter elements to OpenSynapse node types, and creates a draft plan.
  4. Review the import log to see what was converted, approximated, or preserved as code blocks.
  5. The plan opens in the Builder for final adjustments.

These JMeter elements cannot be directly mapped and are preserved as Code Block nodes with comments explaining what was lost:

  • JSR223 samplers with arbitrary Groovy code
  • BeanShell samplers and processors
  • JDBC samplers (planned for v1.1)
  • SOAP samplers (planned for v1.1)
  • JMS samplers (not planned)

Run Export Formats

Every run can be exported in three formats from the Run View toolbar:

  • JSON — Complete run data, re-importable into another OpenSynapse instance.
  • CSV — Time-series metrics in a flat format for spreadsheets and analysis tools.
  • HTML — Standalone report with embedded charts that works offline in any browser.

Troubleshooting / FAQ

The Run button shows "Failed" and doesn't start

Cause: The control plane API is unreachable or the plan has a compilation error.

  • Check that the control plane is running. For Docker, run docker compose ps and verify the control-plane container is healthy.
  • Click "Show Script" in the Plan Builder. If compilation fails, the error message will appear in the script pane.
  • Ensure your plan has at least one Scenario node with at least one sampler child.

Charts show no data during a run

Cause: The WebSocket connection is disconnected.

  • Check the connection indicator in the top-right of the Run View. It should say "Live" with a teal dot.
  • If it says "Disconnected", the browser lost connection to the control plane. Refresh the page to reconnect.
  • If using Docker, ensure the WebSocket port is exposed and not blocked by a reverse proxy.

Live control changes don't take effect

Cause: The plan must use the externally-controlled executor for live control to work.

  • Open the plan in the Builder and check the Scenario node's executor setting.
  • If using a different executor (e.g., constant-vus), VU and RPS controls will send changes but the k6 runtime may not honour them.
  • For full live control support, use the externally-controlled executor with xk6-kv.

JMeter import produces many Code Block nodes

Cause: The .jmx file contains elements that don't have a direct OpenSynapse equivalent.

  • BeanShell, JSR223 Groovy, and JDBC samplers are preserved as Code Blocks with comments.
  • Review each Code Block and rewrite the logic using OpenSynapse's native nodes where possible.
  • Complex Groovy logic may need to be adapted to JavaScript for k6 compatibility.

AI analysis returns an error

Cause: API key issue, network problem, or spend cap reached.

  • Go to Settings > AI and verify your API key is valid.
  • Check that your monthly spend cap hasn't been reached.
  • Ensure your network allows outbound HTTPS connections to your AI provider.
  • Try a different model if the selected one is unavailable.

Crawler doesn't find all pages

Cause: Depth limit, SPA routing, or authentication issue.

  • Increase the crawl depth if your site has deeply nested pages.
  • For SPAs with client-side routing, select the Rod (Browser) engine which uses headless Chromium to render JavaScript — but dynamically loaded content behind user interactions may require a higher request limit. The Colly (HTTP) engine cannot render JavaScript.
  • If pages require authentication, configure the auth settings before crawling.
  • Check the path blocklist — you may have accidentally blocked paths you want to include.

The desktop app won't start

Cause: Port conflict or corrupted database.

  • The control plane binds to port 3000 by default. Check if another process is using that port.
  • On Windows: netstat -ano | findstr :3000
  • On macOS/Linux: lsof -i :3000
  • If the database is corrupted, the app stores its data in the user data directory. You can reset it by deleting the database file (you'll lose existing plans and runs).

Export files are empty

Cause: The run has no metrics data.

  • This happens if the test completed instantly (no requests were made) or if you're exporting a run that's still pending.
  • Wait for the run to reach "completed" or "failed" status before exporting.
  • Verify the plan actually sends requests — a Scenario with no sampler children produces no data.

How do I reset all data?

For the desktop app, delete the SQLite database from your user data directory. For Docker, run:

docker compose down -v
docker compose up -d

This removes all volumes including the database. All plans, runs, and settings will be lost.
